Architecture and platforms.

How the Budget Pacing Tracker is built, where the data flows from, and what to touch (and what not to touch) when maintaining it.

1

Summary

The tracker is a lightweight, serverless application built entirely on Azure and Google Ads' own scripting feature. There's no third-party paid software involved, and no servers to patch or maintain — everything runs on managed platforms already in use elsewhere at FMMS, matching the pattern used for the Jim's Energy Pipeline Dashboard.

2

How Data Flows Through the System

1

Each Google Ads account runs a small script (Google Ads Scripts — a JavaScript feature built into every Google Ads account) on a daily schedule.

2

The script queries that account's campaign cost data using Google Ads Query Language (GAQL), covering the 1st of the month through the end of the previous day.

3

It sends that data as JSON to a dedicated API endpoint, /api/ingest, hosted on Azure.

4

The API checks a shared secret key in the request header (rather than a human login, since a script can't sign in interactively), then merges the new figures into a single stored data record.

5

When someone opens the dashboard, it reads that same stored record, recalculates every campaign's pacing on the fly, and displays it.

6

Anyone using the dashboard directly (adding a campaign, editing a budget) writes back to the same record via a second endpoint, /api/data, which does require a human sign-in.

💡

There are deliberately two separate endpoints with two different authentication methods — one for people (Entra ID login) and one for machines (a shared key) — because Google Ads Scripts has no way to complete an interactive sign-in.

Why This Stack

  • Matches the pattern already used for the Jim's Energy Pipeline Dashboard (Azure Static Web Apps, Entra ID, GitHub Actions), so it's familiar rather than another one-off tool.
  • No new recurring cost of note — Static Web Apps Free tier, a few cents a month of Table Storage for this volume of data, and Google Ads Scripts is free.
  • Nothing to patch, upgrade, or keep alive — all the compute is managed by Azure and Google.

Maintenance Notes

  • Adding a new approved person: update the ALLOWED_EMAILS setting in the Static Web App's Environment variables (Azure Portal).
  • Adding a new Google Ads account: install the script (google-ads-script-template.js, in the repo) in that account, using an account name that either matches one already in the tracker or is a deliberate new one.
  • Rotating the shared automation key: update INGEST_API_KEY in Azure and in every installed Google Ads Script at the same time — they must all match.
  • Changing the dashboard itself: edit index.html in the repository and push to main — GitHub Actions rebuilds and redeploys automatically within a couple of minutes.
🔒

The /api/ingest endpoint is the one intentionally-public route in the app — reachable without an Entra ID login, since Google Ads Scripts can't do interactive sign-in. It's protected instead by the shared key, so treat that key like any other credential.
